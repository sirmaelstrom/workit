## 4. Narrowed Cause

**Root cause:** `buildResults` in service.ts does not guard against `response.results` being `null`. This has always been latent, but the failure rate increased because search-engine is returning `results: null` more often — likely due to an engine configuration change on HOST (engines disabled, quota exhausted, or search-engine updated).

**Line 247 is the crash site, not the root cause.** The fix at 247 is warranted, but the actual root cause lives one level up: search-engine's engine state changed, exposing a pre-existing null guard omission.

**Confidence:** High. Directly reproduced with `results: null` response.

---

## 5. Fix Applied

Minimal fix: guard `response.results` before chaining, with a fallback to empty array.

```typescript
// service.ts, ~line 240
async function buildResults(response: search-engineResponse): Promise<SearchResult[]> {
  const organic = (response.results ?? [])
    .filter(r => r.category === 'general')
    .map(r => ({
      title: r.title,
      url: r.url,
      snippet: r.content,
    }));
  return organic;
}
```

**Why this is the minimal fix:** `?? []` coalesces `null` and `undefined` to an empty array, preserving all existing behavior for valid responses. No interface changes, no logic changes.

**What this does NOT fix:** the `category` field is absent on all results (see Step 5), meaning the filter always returns `[]` regardless. That's a separate functional issue — every search returns zero results even when search-engine responds. But that's not the crash. Fixing both in one edit would exceed scope. The user asked about the crash.

**Bonus finding (not fixing now):** The `category` filter should either be removed, made optional (`r.category === 'general' || !r.category`), or the search-engine query should pass `categories=general` as a param so the field is populated server-side.
