## 3. Evidence Collected

**From knowledge base — architecture context:**

- The synthesis-worker is a **separate pm2 process** added via WP-9 of the synthesis-worker-separation spec. It must have `DISCORD_WEBHOOK_URL` in its own ecosystem env entry — it cannot inherit it from the service process.
- The spec states explicitly: *"same env (needs `DISCORD_WEBHOOK_URL` etc. for briefing delivery)"* — the word "etc." suggests it was noted but may have been copy-pasted incompletely.
- The systems audit flagged a **known silent-failure pattern**: `postWithRetry` only retries 429s. Discord 500s and ECONNRESETs produce no retry and no alert. This means even if the var is present, transient API errors are invisible.
- The health monitor code shows `DISCORD_WEBHOOK_URL || null` — confirming null is a valid runtime value, and the send function likely guards with `if (!url) return` (no error thrown, no log).
- The frontend receiving the briefing proves the **worker → DB → WebSocket path is healthy**. The failure is isolated to the outbound HTTP call to Discord.

## 4. Narrowed Cause

**Most likely: H1 or H4 — `DISCORD_WEBHOOK_URL` absent or null in the synthesis-worker process.**

Confidence: **Medium-high (70%)**

Reasoning:
1. The symptom is a clean split: synthesis works, frontend works, Discord doesn't. This is the Discord send call failing, not a synthesis failure.
2. The synthesis-worker architecture is a recent architectural change where the briefing execution moved to a new process. Env var propagation is the most common failure mode in this class of change.
3. The "silent failure" design means a missing env var produces **zero logs and zero errors** — exactly matching the observed symptom (no Discord message, no apparent error, briefing shows in frontend as if everything worked).
4. The systems audit already called this pattern out as a known flaw.

H6 (incomplete wiring) is the second-most likely at ~20% if the synthesis-worker was recently added via WP-9 and the briefing delivery call wasn't fully wired into the worker's post-completion path.

## 8. Prevention / Memory Capture

Three proposals:

**1. CORRECTIONS.md entry (cross-project):**
> When adding a new pm2 process that is split from an existing one (architecture separation), audit every env var the parent process uses and explicitly list which ones the child needs. "Same env" in a spec is not sufficient — ecosystem.config.js entries are independent and must be explicit. A missing var in a child process silently degrades functionality.

**2. Code fix (silent failure — applies regardless of root cause):**
> `postWithRetry` should surface non-429 HTTP errors with at minimum a `console.error`. The current behavior of silently absorbing Discord 500s and ECONNRESETs makes this class of failure invisible. A log line costs nothing and would have made this diagnosis immediate.

**3. Hook idea (environment audit on pm2 restart):**
> A pre-start hook or a pm2 startup check that validates required env vars are present and non-empty before the process enters the running state. Could be a simple `validateEnv()` function in the worker's `index.ts` that throws with a clear message if `DISCORD_WEBHOOK_URL` is undefined — turning a silent skip into a startup error.
