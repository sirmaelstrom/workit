## 3. Evidence Collected

**From `delivery.ts` (KB-indexed source):**

```typescript
async function getWebhookUrl(
  target: { discord: string },
  _channels: Record<string, string>,
): Promise<string | null> {
  if (target.discord === 'heathdev') {
    return process.env.DISCORD_WEBHOOK_URL || null;  // ← silent null if env var missing
  }
  // ...
}

// In deliverBriefing():
if ('discord' in target) {
  const webhookUrl = await getWebhookUrl(target, channels);
  if (webhookUrl) {
    await postToDiscord(webhookUrl, content);
    console.log(`[Delivery] Posted ${briefingId} to Discord (${target.discord})`);
  }
  // ← NO else, NO log, NO error when webhookUrl is null
}
```

When `getWebhookUrl()` returns null, the entire Discord delivery is silently dropped — no exception, no warning. This is the mechanism that makes the failure invisible.

**From the synthesis-worker separation spec (WP-9):**
> *"Same env (needs `DISCORD_WEBHOOK_URL` etc. for briefing delivery) + `DATA_DIR`"*

This was listed as a requirement but not enforced. pm2's `ecosystem.config.js` env blocks are opt-in per process — if the variable wasn't explicitly copied from the service app entry into the synthesis-worker app entry, it won't be present in `process.env` inside the worker.

**From the synthesis-worker separation review (Round 2, C2 — Worker Environment & Config Gaps):**
> All three models found different facets of the same problem: the worker process spec makes assumptions about environment variables and config structure that don't match the codebase. Rated: Critical.

This was a known risk before dispatch. It is the most likely gap that materialized.

**From the architecture doc:**
> *"The main service process handles scheduling and triggering; the worker executes actual synthesis. IPC uses Postgres LISTEN/NOTIFY."*

Main process registers `frontendBroadcast` via `bridge.ts`. When `handleDispatchComplete` fires in the main process (on receiving the `dispatch_complete` NOTIFY from the worker), it broadcasts directly to frontend. This explains why frontend works regardless of what happens in synthesis-worker — the frontend broadcast path goes through the main process, not through the worker's `deliverBriefing()` call.

The synthesis-worker calls `deliverBriefing()` for `discord` and `file` targets. Discord silently fails. The main process broadcasts to frontend separately. The two paths are independent, which is why the symptom is perfectly split.

## 4. Narrowed Cause

**Root cause: `DISCORD_WEBHOOK_URL` is not present in synthesis-worker's pm2 env block.**

Confidence: **High (85%).**

When synthesis completes, the worker calls `deliverBriefing()`. For the `discord: heathdev` target, `getWebhookUrl()` reads `process.env.DISCORD_WEBHOOK_URL` from the worker's process environment. That variable is absent — it was never forwarded in `ecosystem.config.js` when the synthesis-worker entry was added. The condition `if (webhookUrl)` is false. No Discord POST is made. No error is logged. The worker moves on to file and frontend targets.

frontend still works because the main service process receives the `dispatch_complete` Postgres NOTIFY and runs its own broadcast independently of the worker's delivery call.

The remaining 15% uncertainty is whether delivery was also simply not wired up at all in the worker (H5) — that would produce the same symptom but require a different fix (code wiring vs env config).

## 8. Prevention / Memory Capture

**CORRECTIONS.md entry warranted:**

> **pm2 process env blocks are not inherited — they must be explicitly duplicated.**
>
> When adding a new pm2 process to ecosystem.config.js that shares behavior with an existing process (env vars, secrets, webhook URLs), every required env var must be explicitly listed in the new process's `env` block. pm2 does not inherit env from sibling processes, and it does not load `.env` files unless explicitly configured to do so. Silent failures in code that reads `process.env.X || null` will mask this indefinitely — the code runs fine and produces no errors, the feature just doesn't work.
>
> **Pattern:** Any time a new pm2 process is added for a worker that does I/O (webhooks, API calls, file writes), checklist: copy the full env block from the parent process and trim down from there.

**CLAUDE.md addition for heathdev-service** warranted:

> When adding workers to ecosystem.config.js, diff the env block against the service entry and confirm every secret/webhook URL the worker needs is present. delivery.ts silently skips Discord when `DISCORD_WEBHOOK_URL` is null — no exception is thrown.

**Hook warranted** (build-time check, not session-start):

A `post-ecosystem-edit` check that greps `ecosystem.config.js` for env block completeness across processes. Minimal version: flag if synthesis-worker has fewer env keys than service, since they should share the same secrets.
