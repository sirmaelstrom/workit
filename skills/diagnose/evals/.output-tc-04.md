## 4. Narrowed Cause

**Root cause: The search code path in service is not producing a valid query embedding before running the hybrid search.**

Confidence: **High** (95%).

The DB, index, and stored embeddings are all healthy. A direct vector similarity query against the DB using a real stored embedding as the probe works perfectly. The only remaining explanation for scores of 0.016 is that the query vector passed to `<=>` is a zero vector or null, meaning cosine similarity is undefined/0 for all rows, and the returned score is entirely from the BM25 keyword component.

**Why save works but search doesn't:** These are distinct code paths. `kb_ingest` / `kb_save` embeds at write time. `kb_search` must embed the query at read time. If the embedding call in the search path is silently throwing (and the catch block falls through to keyword-only search), save is unaffected.

**Two equally likely sub-causes:**

1. **The embedding API key for search is missing from service's environment.** If service was restarted (separately from the DB container — the DB restart may be a red herring or misremembered) without its embedding API key env var, the API call fails with a 401 that gets caught and swallowed.

2. **The search function has a silent error path.** The embedding call throws or returns null; the catch block logs at `debug` level (explaining "no errors in service logs" if logs are filtered to `info+`) and falls through to keyword-only search.

## 8. Prevention / Memory Capture

**CORRECTIONS.md entry** (cross-project pattern):

> **Silent embedding fallback in hybrid search**: If the query embedding call fails silently (caught exception, null return, wrong API key), hybrid search degrades to keyword-only and returns uniformly low scores (~0.016) rather than erroring. This is invisible in service logs at INFO level. Symptom is indistinguishable from "no relevant documents" to the end user.
>
> Prevention: log at ERROR level if query embedding returns null before the vector query runs, and surface the embedding model name + vector norm in search debug output.

**CLAUDE.md correction for heathdev-service:**

> When diagnosing kb_search returning low scores (< 0.1): check query embedding generation first, not the DB. The DB vector index has survived multiple container restarts intact. Run `pm2 logs service | grep -i embed` before touching anything else.

**Hook idea** (service health check):
