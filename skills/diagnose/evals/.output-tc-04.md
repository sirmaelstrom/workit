## 4. Narrowed Cause

**Primary suspect: H4 + H1 interaction — missing `VOYAGE_API_KEY` causing silent search failure, compounded by stale pool after restart.**

**Confidence: Medium-high (70%)**

Here is the most coherent narrative:

1. The `heathdev-infrastructure-db` container was restarted.
2. service's pm2 process either (a) was already running and held dead pool connections, or (b) crashed and restarted via the boot-order race.
3. After restart, service re-connected to the DB fine (kb_save works → pool can write).
4. However, `VOYAGE_API_KEY` was not available in the restarted service process — either the pm2 ecosystem config doesn't include it and it was originally sourced from a shell session, or BWS didn't re-inject it on the pm2 restart.
5. When `kb_search` is called, the first step is `embed(query, {input_type: "query"})` via Voyage AI. With no API key, the request returns a 401 or throws a network error.
6. The search handler's catch block returns `[]` (no results) without logging — consistent with "no errors in service logs."
7. `kb_save` appeared to work because either: (a) it was tested after service picked up the key on a subsequent restart, or (b) saves throw loudly on embedding failure (visible to the user) while the search handler swallows it silently.

**Secondary suspect: H1 alone — stale pool, currently self-resolved (confidence: 60%)**

Dead connections in the pg pool would cause search to silently return `[]` if caught. This self-resolves over time as `pg`'s idle timeout evicts dead connections and reconnects. The fact that search works *now* from the MCP tool (which has its own pool, not service's) is consistent with this.

**If content_tsv is empty (H3): High confidence on that sub-path (but would need verification)**

A corrupt or empty `content_tsv` would make FTS return 0. If vector search also fails (H4), both legs of RRF return empty and the FULL OUTER JOIN produces nothing.
