## 4. Narrowed Cause

**Root cause: `pm2 save` was called without bws-injected secrets active, freezing a DISCORD_TOKEN-less env snapshot into `~/.pm2/dump.pm2`. Every subsequent pm2 restart (auto or manual) replays that snapshot.**

Confidence: **High** (95%).

The .env file on disk is a red herring — `ecosystem.config.js` reads `process.env.*` at pm2-start time, not at app-start time, and pm2 does not load .env files unless explicitly configured. Once the bad snapshot is in dump.pm2, no amount of editing the .env file will fix it. The token must be re-injected into pm2's env table by running the proper startup script.

Remaining 5% uncertainty: it's possible the start-services.sh bws call silently failed (expired token, bws not on PATH for the shell running it) — but the behavior would be identical.

---

## 5. Fix Applied

**Do not run `pm2 restart service`.** That replays the bad snapshot.

The correct fix is a full stop-and-start via the bws-wrapped startup script:

```bash
# 1. Stop service
pm2 stop service

# 2. Delete it from pm2's process table (so the stale env is purged)
pm2 delete service

# 3. Re-run the full startup script with bws active
cd /workspace/infrastructure
./start-services.sh       # this runs: eval $(bws secret get ...) && pm2 start ecosystem.config.js && pm2 save

# 4. Verify DISCORD_TOKEN is now set in pm2's env
pm2 show service | grep DISCORD_TOKEN
```

If `start-services.sh` isn't safe to run for all services (service/service are fine), run the targeted equivalent:

```bash
eval $(bws secret get --output env-export)
pm2 start ecosystem.config.js --only service
pm2 save
```

The `pm2 save` after the correct start is critical — it updates dump.pm2 with the correct snapshot so future auto-restarts also have the token.

## 8. Prevention / Memory Capture

**Three things warranted here:**

**A. CLAUDE.md correction (project-level):**
```
Never run `pm2 restart service` directly. Always use the bws-wrapped start script
at /workspace/infrastructure/start-services.sh, or run `eval $(bws secret get ...)` first.
Direct pm2 restarts replay the env snapshot without re-fetching secrets.
```

**B. CORRECTIONS.md entry (cross-project — this pattern will bite any bws+pm2 service):**
```
pm2 + bws: pm2 stores env at start time (not restart time). `pm2 save` freezes that snapshot.
If bws was not active when pm2 started, DISCORD_TOKEN (and other bws-sourced vars) will be
undefined in every subsequent restart including auto-restarts after crashes.
Fix: pm2 delete <name> → eval $(bws secret get ...) → pm2 start → pm2 save.
Symptoms: env var undefined despite .env file existing on disk.
```

**C. Hook idea:** Add a post-start hook or a pm2 pre-startup script that validates `DISCORD_TOKEN` is non-empty and halts with a clear error rather than letting service start in a broken state. A startup guard like:

```js
// near top of index.ts
if (!process.env.DISCORD_TOKEN) {
  console.error('[service] FATAL: DISCORD_TOKEN is not set. Was bws active at pm2 start?');
  process.exit(1);  // fail fast: pm2 marks errored, doesn't silently limp
}
```

This surfaces the root cause immediately instead of letting the app start and fail on first Discord API call.
