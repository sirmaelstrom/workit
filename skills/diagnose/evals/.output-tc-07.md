## 4. Narrowed Cause

**Root cause: H1 — pm2 auto-restart bypassed the bws secrets injection layer.**

**Confidence: High (95%)**

When service crashed, pm2's crash-recovery kicked in and restarted the process using the command it has cached internally (`node dist/server.js`). The original launch was done via a shell wrapper or `bws run` prefix that injected secrets into the process environment before starting node. pm2 never stored those outer-wrapper details — it only stored the inner command.

Result: the process comes back online, passes pm2's health check, but runs with no BWS-injected environment variables. The .env file on disk is irrelevant if the project uses `bws run` as its secrets layer and dotenv is either not loaded, loaded after bws, or not used at all in production.

This is the known "pm2-without-secrets" failure pattern, referenced explicitly in this skill's anti-patterns as a prior incident.

---

## 5. Fix Applied

**Do not just run `pm2 restart service`.** That will restart with the same bare command and the same empty env. You'll be right back here.

**Correct fix: kill the pm2 entry and re-launch through the official start script.**

```bash
# Step 1: stop and delete the bare process entry
pm2 delete service

# Step 2: re-run the official start script (this calls bws run, injects secrets, then starts node)
# Adjust path to match your actual start script location:
bash /workspace/infrastructure/start-service.sh
# OR if bws wraps directly:
bws run --access-token $BWS_ACCESS_TOKEN -- pm2 start ecosystem.config.js --env production

# Step 3: verify env was injected
pm2 env service | grep DISCORD_TOKEN

# Step 4: save the pm2 process list so resurrect picks up the right entry
pm2 save
```

**Why this is the minimal fix:** We're not changing any code. We're restoring the process to the state it was in before the crash — running under the secrets-injected environment the start script provides.

**What NOT to do:**
- Don't hardcode `DISCORD_TOKEN` into ecosystem.config.js — that puts secrets in source control
- Don't `pm2 restart` — same root cause, same outcome
- Don't rely on pm2's `--update-env` flag — it only picks up env block values from the ecosystem file, not bws-injected values

## 7. Remaining Uncertainty

1. **Why did service crash in the first place?** The secrets issue is the restart symptom, not the crash cause. Worth checking `pm2 logs service --err` for the original crash reason — it may indicate a deeper instability.

2. **Is pm2 resurrect/startup configured to use the start script?** If the machine reboots, `pm2 resurrect` will reload the saved process list — which may again be the bare `node` command without bws. The startup/resurrect path needs the same secrets injection or it'll fail again on reboot.

3. **Is the .env file used at all?** If the project loads dotenv and bws *both*, there may be a precedence issue. Worth confirming which layer wins and whether the .env is dev-only or also present in production.

4. **bws token validity:** The `BWS_ACCESS_TOKEN` used to call bws must still be valid. If it expired between the original launch and now, the start script will fail silently or with a bws auth error.

## 8. Prevention / Memory Capture

### CORRECTIONS.md entry (cross-project, high recurrence risk)

> **pm2 auto-restart does not re-inject bws secrets.**
> When a pm2-managed service crashes and auto-restarts, pm2 uses its cached command (`node dist/server.js`), not the outer shell wrapper or `bws run` prefix that was used at original launch. The restarted process has an empty environment — all bws-injected secrets are absent. Symptoms appear as "env var undefined" errors rather than crash-on-start, making the root cause non-obvious.
>
> **Rule:** After any unexpected pm2 restart of a bws-managed service, always verify `pm2 env <name>` before debugging the application layer. Always re-launch via the start script, never via `pm2 restart`.

### ecosystem.config.js hardening idea

Consider adding a pre-start hook in ecosystem.config.js that validates critical env vars exist and exits non-zero (preventing pm2 from marking the process online) if they're absent:

```js
// At the top of server.js or as a pre-start check
const required = ['DISCORD_TOKEN', 'DATABASE_URL'];
const missing = required.filter(k => !process.env[k]);
if (missing.length) {
  console.error(`[startup] FATAL: missing env vars: ${missing.join(', ')}`);
  console.error('[startup] This process was likely restarted without bws injection. Use the start script.');
  process.exit(1);
}
```

This makes the failure loud and immediate (pm2 shows "errored" rather than "online") instead of silently running a crippled process.

### Hook idea

A session-start hook for service work sessions could run `pm2 env service | grep -E "DISCORD_TOKEN|DATABASE_URL"` and surface a warning if critical secrets are absent — catching this before the user tries to trigger a briefing.
