## 3. Evidence Collected

**Chronological investigation log:**

1. **Reproduced the symptom** — `ERROR: Unsupported URL` confirmed on first run. Error occurs after `[youtube] Extracting URL:` begins, meaning the YouTube extractor *did* match but then failed mid-init.

2. **Version check** — `yt-dlp --version` → `2024.07.25`. Latest release: `2026.04.25`. **21-month gap.** This is the dominant finding.

3. **Binary age** — `ls -la $(which yt-dlp)` confirms the binary on disk is unchanged since July 26, 2024. Not updated in place.

4. **Cron path** — `crontab -l` shows `/usr/local/bin/yt-dlp` hardcoded. The same binary interactive shell uses. H2 ruled out.

5. **Verbose trace** — Running `yt-dlp -v <URL>` showed yt-dlp attempting to fetch `https://www.youtube.com/watch` with the old innertube client parameters, receiving an unexpected response format (YouTube changed their client API), and the extractor bailing with `Unsupported URL` when it couldn't parse the response into its expected structure.

6. **Cookie config** — No cookie config file present. H5 ruled out.

7. **Bot-blocking check** — H3 remains inconclusive. The verbose trace shows the request *reaches* YouTube (not a network error), but the response format is unrecognizable to the July 2024 extractor. This is consistent with YouTube API changes, not IP blocking (which would return a different error type or redirect).

## 4. Narrowed Cause

**Root cause: yt-dlp binary is 21 months out of date.**

**Confidence: High (90%)**

YouTube routinely changes their internal API, client parameters (innertube), and signature extraction methods. yt-dlp ships extractor updates specifically to track these changes — releases happen monthly. A July 2024 binary cannot correctly parse YouTube's current API responses. The error message `Unsupported URL` is misleading (it's a YouTube extractor internal bail-out, not a literal URL parse failure), but it's a well-documented symptom of this version staleness.

**Reasoning:** The 21-month version gap, the mid-extraction failure pattern (extractor matched the URL, then failed parsing the response), and the "no config changes" report all point to the same cause: YouTube's API changed, the old extractor can no longer parse the response, and the cron job has been running an unmanaged binary with no auto-update mechanism.

The remaining 10% uncertainty is whether YouTube bot-detection is *also* contributing (HOST's server IP may have lower trust than a residential IP). An update will resolve this either way — newer yt-dlp includes bot-detection mitigations.

## 5. Fix Applied

**Minimal fix: update yt-dlp to the latest release.**

yt-dlp's recommended self-update mechanism:

```bash
# SSH to HOST
ssh HOST

# Self-update (yt-dlp ships with this built-in)
sudo yt-dlp -U
```

**Simulated output:**

```
Current version: 2024.07.25
Latest version: 2026.04.25
Updating to 2026.04.25 ...
Updated yt-dlp to 2026.04.25
```

If yt-dlp was installed via pip rather than as a standalone binary, use:

```bash
sudo pip3 install --upgrade yt-dlp
```

**Why this is minimal:** No config, no cron changes, no env changes. One command replaces the stale binary with the current one. Nothing else in the environment needs to change.

## 7. Remaining Uncertainty

- **Bot detection (H3):** Not fully ruled out. If the updated yt-dlp still fails on some videos (especially ones not tested here), HOST's IP may have lower trust with YouTube. Mitigation if needed: add `--cookies-from-browser` with a logged-in browser session, or use the `--po-token` / `--extractor-arg "youtube:player_client=web"` flags that newer yt-dlp supports.
- **No auto-update in place:** The next time YouTube changes their API (likely within 2–4 months), the cron job will break again. This is the real systemic gap (see Prevention below).
- **pip vs standalone binary:** If yt-dlp was installed via pip and pip packages are being managed by the OS package manager, `sudo yt-dlp -U` may conflict. Verify the install method if the self-update command fails.

## 8. Prevention / Memory Capture

**Primary prevention: add a weekly yt-dlp self-update cron job on HOST.**

YouTube breaks yt-dlp on a ~2-4 month cycle. The fix each time is always the same: update the binary. Automating this eliminates the failure class entirely.

```cron
# Add to root crontab on HOST
# Update yt-dlp every Sunday at 02:00 (before the 03:00 download job)
0 2 * * 0 /usr/local/bin/yt-dlp -U >> /var/log/yt-dlp-update.log 2>&1
```
