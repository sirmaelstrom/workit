**From briefing daily logs (2026-04-23):**

> *"Briefing retry storm — fixed once, still insufficient. Apr 21 was 155 lines of synthesis failures — 5.5 hours of wasted compute overnight. Circuit breaker (30-min cooldown) lacks escalating backoff or daily cap."*

**Key structural fact confirmed from KB:**

The briefing watchdog is **self-referential** — it runs on the same node-cron scheduler it monitors. When the scheduler wedges, the watchdog also stops ticking. This is why 6 hours can pass without an alert.

## 4. Narrowed Cause

**H1 — node-cron scheduler wedge — confidence: HIGH**

The symptom profile is a precise match to a documented prior incident:

| Signal | Wedge scenario | This incident |
|--------|---------------|---------------|
| pm2 status | `online`, 0 restarts | `online`, 0 restarts |
| Briefing output | Silent for hours | Silent for ~6 hours |
| Code changes | None required | None since yesterday |
| Watchdog alert | Never fires (also wedged) | No alert received |

The mechanism: node-cron's internal `setInterval` runs on the Node.js event loop. When a prior briefing synthesis (Claude CLI child process, potentially multi-turn Opus) created sustained I/O callback pressure, cron's ticks were delayed indefinitely. The process looks healthy because the Node process is alive — the event loop is just saturated or stuck behind a never-resolved promise or hung child process.

**H2 (stuck subprocess) may be co-occurring** — a hung Claude CLI process from the last successful synthesis run would explain why the event loop never cleared. Worth checking before restarting.

## 5. Fix Applied

**Step 1 — Inspect before restarting (non-destructive):**

```bash
# Check for hung claude subprocess
tasklist /fi "imagename eq claude.exe" 2>&1

# Check pm2 ecosystem config to confirm start script is correct
pm2 show service

# Confirm uptime (how long has it been running without restart?)
pm2 list
```

**Step 2 — If hung subprocess found, kill it first:**

```bash
# Kill hung claude processes before restarting service
taskkill /f /im claude.exe
```

**Step 3 — Restart service:**

```bash
pm2 restart service
```

This reinitializes node-cron from scratch, re-registers all cron jobs, clears any wedged setInterval state, and reloads BWS secrets.

**Step 4 — Confirm next scheduled briefing fires:**

```bash
# Watch logs for the next cron tick (should see within 1-2 minutes for maintenance briefing)
pm2 logs service --lines 50
```

**Do NOT** use `pm2 delete service && pm2 start ...` — the ecosystem file is the canonical start path, and recreating the process entry risks losing env vars or using the wrong start script.
