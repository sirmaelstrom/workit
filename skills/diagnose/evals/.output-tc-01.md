## 4. Narrowed Cause

**Root cause: retired Claude model string in `synthesis.ts`**  
**Confidence: High**

`claude-sonnet-4-5-20251001` was retired from the Anthropic API. The Claude CLI rejects it with exit code 1 and writes the error to stderr — but service's synthesis runner doesn't capture stderr, so the error is invisible. The circuit breaker then correctly exhausted its daily retry cap (4 cycles), which is why delivery is now completely silent despite the process being healthy.

Two compounding factors:
1. **No stderr capture** — the retirement error is swallowed. Without it, the circuit breaker is the only observable signal, and it looks like "broken, then quiet."
2. **Circuit breaker daily cap** — correct behavior given the root cause, but masks the failure by ending retry attempts before anyone would naturally look at the logs.

## 5. Fix Applied

**Fix 1 (immediate, ~1 minute):** Update the model string in `synthesis.ts`.

```typescript
// src/briefings/synthesis.ts  — before
model: 'claude-sonnet-4-5-20251001',

// after
model: 'claude-sonnet-4-6',
```

**Fix 2 (required to unblock today):** Reset the circuit breaker state so deliveries resume before midnight. The daily counter in `briefing-exec.ts` resets at midnight, so without a restart, today's missed briefings stay missed.

```bash
pm2 restart service
```

This is the correct action here — the restart is warranted because:
- The code fix needs to be picked up (process doesn't hot-reload)
- The circuit breaker daily counter resets on process restart
- Secrets will be re-fetched from BWS at startup (no concern — the API key isn't stale)

**What was NOT changed:** No retry logic, no cron config, no secrets, no infrastructure. Minimal surface area.

## 8. Prevention / Memory Capture

**CORRECTIONS.md entry:**

> **Never hardcode a model ID with a date suffix in a long-running service.**
> `claude-sonnet-4-5-20251001` retired on ~2026-04-29, silently killing briefing synthesis with exit code 1.
> Use a version-stable alias (`claude-sonnet-4-6`) or read the model string from a config/env var that can be updated without a deploy.
> **service:** `src/briefings/synthesis.ts`, `model:` field.

**LESSONS.md entry:**

> **Subprocess stderr must be captured for exit-code-1 debugging.**
> service's synthesis runner swallows Claude CLI stderr. When the CLI fails (retired model, auth error, rate limit), you get `exit code 1` and nothing else. The fix is a one-liner: pipe stderr to a buffer and include it in the error log.
> Pattern: `const result = spawnSync(cmd, args, { encoding: 'utf8', stdio: ['pipe','pipe','pipe'] }); if (result.status !== 0) log.error('stderr:', result.stderr);`

**Hook idea:**

> Add a cron-checked alert: if `briefing_runs` has no `status = 'success'` row in the past 2 hours during daytime hours (6am–10pm), fire a Discord notification to the assistant webhook. This is a liveness check independent of the circuit breaker — the circuit breaker going silent is indistinguishable from healthy silence without this.

**CLAUDE.md project correction (service):**

> When updating the Claude model version in service, always update `src/briefings/synthesis.ts` AND verify the model alias is not a dated ID. Use the service CLAUDE.md "build before commit" reminder as the checkpoint.
