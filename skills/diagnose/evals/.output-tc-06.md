## 4. Narrowed Cause

**Primary suspect: H3 — Time-based rotation is checked on message arrival, not by a background timer.**

Confidence: **Medium-High** (70%).

Reasoning: The stated constraint — "not on a fixed schedule" — is the key tell. If rotation were driven by a background poll (setInterval every hour), it would be on a fixed schedule. The constraint language is specifically ruling that out. This implies the time threshold is evaluated lazily: when a new message arrives, service checks if the session is too old or too long before processing it — and rotates if so. A session that has sat dormant for 3 days with **no new messages** would never have triggered that check.

This is not a bug — it's a design consequence the user may not have internalized. The session "should have rotated" only if something was supposed to trigger the check, which requires either:
- A new message sent through frontend
- A manual `session.rotate` WebSocket call
- A background scheduler in the codebase (not observed)

**Secondary suspect: H1 — `maxAgeHours` is larger than the user believes.** If the threshold is 72h or 96h (not 24h), a 3-day session might be legitimately within bounds. The architecture doc with the wrong message count (40 vs 60) suggests threshold values in docs can drift.

**Tertiary suspect: H4 — The session did rotate and the user missed it.** Worth ruling out cheaply before doing anything else.
