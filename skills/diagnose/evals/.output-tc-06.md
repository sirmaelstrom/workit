## 4. Narrowed Cause

**Primary cause: H3 — lazy rotation, message-triggered only.**

**Confidence: High (85%)**

The rotation mechanism is demand-driven. The 24-hour age check exists inside `getOrCreatefrontendSession()`, which only runs when a new message arrives. After 3 days of no activity, that code path has never executed — not because it's broken, but because nothing called it.

The phrase "it should have rotated by now" reflects a reasonable but incorrect mental model: that the 24-hour threshold triggers proactively. It doesn't. Time is a rotation *condition*, not a *signal*. The signal is always an incoming message.

**Secondary concern: H5 — `startedAt` reset on service restart (confidence: Low-Medium)**

service is pm2-managed and has been restarted multiple times (March debugging sessions). If service restores an active session row from the DB on startup but updates `startedAt` to restart time rather than the original creation time, the age check would always see a "fresh" session. This would be a silent correctness bug that only surfaces during long idle periods.

---

## 5. Fix Applied

No code change applied — the primary cause is a design characteristic, not a defect.

**Immediate action:** Send any message to frontend. `getOrCreatefrontendSession()` will run, evaluate `session.startedAt` age against `MAX_SESSION_AGE_MS`, find it 72h+ over threshold, and trigger rotation.

**If rotation does NOT fire after sending a message**, escalate to H5 investigation:
1. Check the `sessions` table for the current session's `started_at` column
2. Compare against service's pm2 `startTime` — if they match, `startedAt` is being reset on restart
3. Fix: restore `startedAt` from the existing DB row on startup, do not overwrite it

## 8. Prevention / Memory Capture

**A. CORRECTIONS.md entry (cross-project pattern):**

> **"Time threshold" ≠ "time-based trigger"** — A condition like `age > 24h` only fires if something evaluates it. In service, session rotation is demand-driven (message intake); the time threshold is a *condition*, not a *signal*. If a service can be idle for longer than its threshold, it needs a background probe (setInterval, cron, or pm2 hook) to enforce the bound proactively — the condition alone is not enough.

**B. service CLAUDE.md correction:**

Add to session rotation section:
> Session rotation is **message-triggered only**. The 24-hour age threshold is evaluated inside `getOrCreatefrontendSession()` — no background timer exists. A session idle longer than 24 hours will not auto-rotate until the next message arrives. To add proactive rotation, add a `setInterval` in bridge startup that calls the same rotation path every ~30 minutes.

**C. Feature suggestion — proactive rotation timer:**

Add to `bridge.ts` startup:
```typescript
// Check session age every 30 minutes; rotate if over MAX_SESSION_AGE_MS
setInterval(async () => {
  const session = await getActiveSession(frontend_CHANNEL_ID);
  if (session && shouldRotate(session)) {
    await rotateSession(session, 'timer');
  }
}, 30 * 60 * 1000);
```

This closes the gap permanently and makes the "24-hour rotation" contract actually hold during idle periods.

---

**Bottom line:** The session almost certainly isn't broken — it just hasn't had a chance to check its own age. Send a message and it will rotate. The real issue is that "time-based rotation" implies something watching the clock, but the current implementation only runs the clock check reactively. The timer-based probe in item C above is the correct long-term fix.
