## 4. Narrowed Cause

Without running the tests, two hypotheses lead on probability:

**Primary suspect — H1 (stale Vite/HMR state), confidence: 60%**
Rationale: "Was working yesterday, nothing changed" is the exact fingerprint of HMR state corruption in Tailwind 4 + Vite. Tailwind 4 processes CSS through Vite's transform pipeline at build time and via HMR updates; after a long dev session with many saves, the emitted CSS can fall out of sync with the actual Tailwind class scan. A hard dev server restart almost always resolves this. This is not a code bug — it's a build toolchain state issue.

**Secondary suspect — H3 (DaisyUI version drift), confidence: 20%**
Rationale: DaisyUI 5 changed `.card`'s internal layout model (moved from flexbox to grid in some releases during the 5.x beta cycle). A silent `npm install` or lockfile change could have upgraded it. If the card component's layout depends on `.card`'s internal flex/grid direction, a version bump would break it silently with "no CSS changes."

**Tertiary — H2 (Tailwind class purge), confidence: 10%**
Rationale: If the parent container's `grid` or `flex` class is constructed dynamically, Tailwind 4's scanner might have missed it. However, this tends to produce a consistent failure rather than a regression, making it less likely for a "was working yesterday" report.
